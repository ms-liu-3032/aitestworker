# 10 轨迹摘要文本化 P0 设计与实现方案

## 1. 背景与目标

当前仓库已经具备结构化轨迹采集、轨迹清洗、轨迹详情展示、轨迹摘要生成雏形、测试用例草稿生成、Skill / Tool 提炼、受控扫描与页面画像等基础能力，但现有输出仍偏“操作流水”。

本方案的目标是在现有结构化轨迹能力之上，新增一层：

**轨迹摘要文本化（Trace Summary Textualization）**

将：

- `trace_group`
- `trace_session`
- `trace_event`
- `trace_network`
- `trace_issue_clip`

转换为测试人员、产品和开发都能理解的**业务流程摘要草稿**。

该层的定位是：

1. 轨迹详情页的增强能力；
2. 测试用例草稿生成的上游输入；
3. Mini-TOM 候选抽取的上游输入；
4. 非正式资产层，默认仅为草稿，不自动进入正式知识库。

### 1.1 明确边界

本方案明确**不做**：

1. 视频理解；
2. 截图 OCR；
3. 截图语义识别；
4. 新的视觉模型；
5. 自主 Agent；
6. 自动写入正式知识库；
7. 自动写入 Weaviate；
8. 自动写入 Neo4j；
9. 未确认摘要直接进入 Mini-TOM。

### 1.2 核心原则

1. 摘要只基于结构化轨迹数据生成；
2. 摘要默认 `DRAFT`；
3. 未确认摘要不得进入 Mini-TOM；
4. 未确认摘要不得进入 Weaviate / Neo4j / 正式知识资产；
5. 必须坚持“候选 / 草稿 → 人工确认 → 正式沉淀”。

## 3. P0 范围定义

### 3.1 P0 要做

P0 只做以下能力：

1. 基于结构化轨迹生成固定格式摘要草稿；
2. 支持按 `GROUP / SESSION / ISSUE_CLIP` 范围生成；
3. 支持重新生成；
4. 支持编辑；
5. 支持确认有效；
6. 支持驳回；
7. 支持生成来源追踪；
8. 支持与测试用例生成链路衔接；
9. 支持明确门禁，阻止未确认摘要进入 Mini-TOM / Weaviate / Neo4j。

### 3.2 P0 不做

P0 不做以下内容：

1. 视频内容理解；
2. 截图 OCR；
3. 截图智能识别；
4. 视觉模型；
5. 自主 Agent；
6. 自动写入正式知识库；
7. 自动写入 Weaviate；
8. 自动写入 Neo4j；
9. 自动写入正式 Mini-TOM；
10. 自动推导完整功能覆盖。

## 4. `trace_summary` 数据结构设计

### 4.1 设计原则

1. 尽量兼容现有表结构；
2. 优先 `ALTER`，避免无意义重建；
3. 保留旧字段用于兼容或过渡；
4. 以结构化字段为主，以 `summary_text` 为兼容全文字段。

### 4.2 推荐字段

#### 关联字段

- `id`
- `project_id`
- `trace_group_id`
- `trace_session_id`，可空
- `issue_clip_id`，可空
- `summary_scope`：`GROUP / SESSION / ISSUE_CLIP`

#### 摘要正文

- `overview`
- `business_summary`
- `key_steps_json`
- `key_api_json`
- `exception_summary`
- `case_generation_suggestion_json`

#### 生成快照

- `raw_input_snapshot`
- `llm_output_snapshot`
- `model_config_id`
- `prompt_snapshot_id`
- `context_manifest_id`
- `llm_invocation_log_id`

#### 状态治理

- `status`：`DRAFT / REVIEWING / CONFIRMED / REJECTED / DEPRECATED`
- `validity_label`：
  - `STANDARD`
  - `PERSONAL_ONLY`
  - `TASK_ONLY`
  - `BUG_REPRO`
  - `DEMO`
  - `DEBUG_DATA`
  - `DIRTY_DATA`
  - `OLD_VERSION`
  - `TO_CONFIRM`

#### 审核字段

- `created_by`
- `confirmed_by`
- `confirmed_at`
- `rejected_by`
- `rejected_at`
- `rejected_reason`
- `deprecated_by`
- `deprecated_at`
- `deprecated_reason`
- `created_at`
- `updated_at`

#### 置信度与待确认

- `confidence_label`：`HIGH / MEDIUM / LOW`
- `pending_confirmation_json`

### 4.3 现有字段保留策略

现有字段建议保留：

- `summary_text`
- `scope`
- `review_status`
- `trust_level`

建议用法：

1. `summary_text` 作为全文拼接或兼容展示；
2. `scope` 逐步映射到 `summary_scope`；
3. `review_status` 过渡期兼容旧逻辑；
4. `trust_level` 保留为治理辅助字段。

### 4.4 建议索引

- `(trace_group_id, status)`
- `(trace_session_id)`
- `(issue_clip_id)`
- `(project_id, created_at)`
- `(status, validity_label)`

## 5. 摘要输入与输出规范

### 5.1 摘要输入

摘要仅允许使用以下结构化输入：

1. `trace_group`
2. `trace_session`
3. `trace_event`
4. `trace_network`
5. `trace_issue_clip`
6. `browser_profile / role_label`
7. `url`
8. `page_title`
9. `element_text`
10. `element_role`
11. `value_summary`
12. `normalized_locator`
13. `section_title`
14. `dialog_title`
15. `object_label`
16. `method / request_url / status / duration_ms`
17. `local_time / utc_time / relative_ms`

### 5.2 禁止输入

明确禁止：

1. 视频内容；
2. 截图内容；
3. OCR；
4. 视觉模型推断。

### 5.3 固定输出结构

每次摘要生成必须输出以下 5 段：

#### 【轨迹概述】

用 1 句话说明本次轨迹记录的业务流程。

#### 【业务流程摘要】

用 3-6 句话描述用户完成的业务动作、涉及角色、页面和结果。

#### 【关键操作步骤】

用编号列出关键步骤，不包含过细的点击流水。

#### 【关键接口与异常】

说明关键接口、状态码、失败请求、慢请求或异常请求。  
若无明显异常，必须输出：

- `未发现明显异常请求`

#### 【用例生成建议】

适合生成：
- xxx

不适合生成：
- xxx

原因：
- xxx

---

## 6. 摘要质量规则

摘要生成必须遵循以下规则：

1. 不输出机械点击流水；
2. 不只写“点击了什么、输入了什么”；
3. 要尽量转成业务动作；
4. 不允许编造未发生的流程；
5. 不允许扩展未发生的业务分支；
6. 多角色轨迹必须标明角色；
7. 单角色轨迹必须明确“仅覆盖该角色视角”；
8. 如果只覆盖主流程，必须说明不适合生成完整功能覆盖用例；
9. 如果存在 4xx / 5xx / 超时请求，必须在异常部分说明；
10. 如果有 `issue_clip`，必须优先说明问题发生前后的关键操作；
11. 摘要默认状态为 `DRAFT`；
12. 未确认摘要不得进入 Mini-TOM；
13. 未确认摘要不得进入 Weaviate；
14. 未确认摘要不得进入 Neo4j；
15. 未确认摘要不得影响其他用户后续生成。

## 7. 置信度分层与待确认策略

### 7.1 高置信内容

以下内容可以直接进入摘要正文：

1. 页面跳转明确；
2. 主提交流程明确；
3. 成功 / 失败提示明确；
4. 网络请求与动作关系明确；
5. 角色身份明确；
6. issue clip 覆盖范围明确。

### 7.2 中置信内容

以下内容可以进入摘要，但必须使用保守表达：

1. 选择了某类业务对象；
2. 执行了某个详情 / 展开 / 修改类动作；
3. 勾选了某个通知选项；
4. 进入了某个确认弹窗并完成确认。

示例表达：

- “用户在历史记录中选择了业务对象”
- “用户在取消邀请弹窗中选择了通知选项”

### 7.3 低置信 / 待确认内容

以下内容不得直接写成稳定事实，应记录到：

- `pending_confirmation_json`

典型情况：

1. 行对象文本异常；
2. checkbox 业务含义不稳定；
3. 连续 click 无法稳定映射；
4. 输入法中间态污染字段；
5. 行对象、图标按钮、弹窗按钮发生混淆。

### 7.4 `pending_confirmation_json` 建议结构

```json
[
  {
    "type": "OBJECT_LABEL",
    "field": "history_object",
    "value": "0.87358400113361119468794544",
    "reason": "对象识别疑似被错误字段污染"
  },
  {
    "type": "ACTION_SEMANTICS",
    "field": "cancel_invite_option",
    "value": "勾选确定",
    "reason": "checkbox 业务语义不稳定"
  }
]
```

## 8. 后端服务设计

### 8.1 服务名称

建议新增正式服务：

- `TraceSummaryService`

### 8.2 服务方法

#### `generateSummary(traceGroupId, scope)`

输入：

- `traceGroupId`
- `summaryScope`
- `traceSessionId?`
- `issueClipId?`
- `modelConfigId`

输出：

- 新生成的 `trace_summary` 草稿

行为：

1. 读取轨迹范围；
2. 调用清洗器；
3. 组装 Prompt 输入；
4. 调用 `LlmGateway`；
5. 落 `trace_summary`；
6. 写快照与调用记录引用。

#### `regenerateSummary(summaryId)`

输入：

- `summaryId`

输出：

- 新版本摘要草稿或覆盖当前草稿

行为：

1. 读取原摘要 scope；
2. 重建原始输入；
3. 重新调用模型；
4. 更新草稿与快照引用。

#### `updateSummary(summaryId, payload)`

输入：

- `summaryId`
- 用户编辑后的结构化内容

输出：

- 更新后的摘要

可编辑内容：

- `overview`
- `business_summary`
- `key_steps_json`
- `key_api_json`
- `exception_summary`
- `case_generation_suggestion_json`
- `validity_label`
- `pending_confirmation_json`

#### `confirmSummary(summaryId, validityLabel)`

输入：

- `summaryId`
- `validityLabel`

输出：

- 已确认摘要

行为：

1. 校验权限；
2. 写 `status=CONFIRMED`；
3. 写 `confirmed_by / confirmed_at`；
4. 保留原快照引用。

#### `rejectSummary(summaryId, reason)`

输入：

- `summaryId`
- `reason`

输出：

- 已驳回摘要

行为：

1. 写 `status=REJECTED`；
2. 写 `rejected_by / rejected_at / rejected_reason`。

#### `getSummary(traceGroupId)`

输入：

- `traceGroupId`

输出：

- 该 group 下摘要列表或默认摘要

## 9. API 设计

建议新增以下 API：

### 9.1 生成摘要

`POST /api/trace/groups/{groupId}/summaries:generate`

请求体：

- `modelConfigId`
- `summaryScope`
- `traceSessionId?`
- `issueClipId?`

### 9.2 获取 group 摘要列表

`GET /api/trace/groups/{groupId}/summaries`

### 9.3 获取单条摘要

`GET /api/trace/summaries/{summaryId}`

### 9.4 重生成摘要

`POST /api/trace/summaries/{summaryId}:regenerate`

### 9.5 更新摘要

`PUT /api/trace/summaries/{summaryId}`

### 9.6 确认摘要

`POST /api/trace/summaries/{summaryId}:confirm`

请求体：

- `validityLabel`

### 9.7 驳回摘要

`POST /api/trace/summaries/{summaryId}:reject`

请求体：

- `reason`

## 10. LLM Prompt 设计

### 10.1 System Prompt 原则

System Prompt 必须明确：

1. 只能基于真实结构化轨迹；
2. 不得补未发生分支；
3. 必须按固定结构输出；
4. 输出中文；
5. 输出自然、清楚、专业；
6. 若对象或动作不够确定，应保守表达；
7. 不得把摘要写成点击流水。

### 10.2 User Prompt 组成

建议由以下块组成：

1. 轨迹范围说明；
2. 角色与身份空间说明；
3. 页面画像上下文；
4. 清洗后步骤；
5. 网络时间线；
6. issue clip 片段；
7. 待确认项提示；
8. 固定输出格式要求；
9. 用例生成建议边界要求。

### 10.3 推荐输出 JSON

建议先让模型输出结构化 JSON，再由后端渲染为固定文本结构。

推荐 JSON：

```json
{
  "overview": "...",
  "businessSummary": ["...", "..."],
  "keySteps": ["...", "..."],
  "keyApis": [
    {
      "method": "POST",
      "url": "/api/...",
      "status": 200,
      "remark": "..."
    }
  ],
  "exceptionSummary": "...",
  "caseGenerationSuggestion": {
    "fit": ["..."],
    "unfit": ["..."],
    "reasons": ["..."]
  },
  "pendingConfirmation": [
    {
      "type": "OBJECT_LABEL",
      "value": "...",
      "reason": "..."
    }
  ]
}
```

## 12. 与测试用例生成链路的关系

### 12.1 旧链路

结构化轨迹  
→ 直接生成测试用例草稿

### 12.2 新链路

结构化轨迹  
→ 轨迹摘要文本化  
→ 用户确认或标记待确认  
→ 基于摘要生成测试用例草稿  
→ 用户本地用例库

### 12.3 规则

1. 已确认摘要优先作为生成依据；
2. 未确认摘要可用于当前任务临时生成，但必须标记“待确认摘要生成”；
3. 未确认摘要生成的测试用例不得自动进入正式库；
4. 测试用例草稿不得自动写入正式用例库；
5. 用户导出或手动提交后，测试用例才进入正式库。

---

## 13. 与 Mini-TOM 的门禁设计

### 13.1 基本门禁

1. 只有 `status = CONFIRMED` 的 `trace_summary` 才允许进入 Mini-TOM 候选抽取；
2. `DRAFT / REVIEWING / REJECTED / DEPRECATED` 的 `trace_summary` 不允许进入 Mini-TOM；
3. Mini-TOM 候选仍需二次人工确认；
4. 未确认 Mini-TOM 不得进入正式资产；
5. 未确认 Mini-TOM 不得参与默认生成。

### 13.2 进一步建议

如果 `pending_confirmation_json` 仍不为空，则：

1. 可进入 Mini-TOM 候选草稿层；
2. 但必须标记为 `TO_CONFIRM`；
3. 不允许进入正式资产或默认生成链路。

3. 保持旧代码仍可读现有摘要记录；
4. 新代码逐步切到结构化字段。

## 16. 验收标准

P0 完成后必须满足：

1. 能基于单个 `trace_group` 生成固定格式摘要；
2. 摘要不是机械点击流水；
3. 摘要能说明用户完成的业务动作；
4. 摘要能说明涉及角色和页面；
5. 摘要能说明轨迹覆盖范围；
6. 摘要能说明不适合生成哪些用例；
7. 摘要能识别异常请求；
8. 用户可以重新生成摘要；
9. 用户可以编辑摘要；
10. 用户可以确认摘要有效；
11. 用户可以驳回摘要；
12. 未确认摘要不能进入 Mini-TOM；
13. 未确认摘要不能写入 Weaviate / Neo4j；
14. 摘要生成有 `prompt_snapshot` 和 LLM 调用记录；
15. 基于摘要生成的用例仍然只进入用户本地用例库。

---

## 17. 风险点

1. 当前结构化语义仍不完全稳定，尤其是：
   - 行对象识别
   - checkbox 语义
   - 弹窗确认链路
2. 现有摘要链路还在 `knowledge_asset`，切到 `trace_summary` 需要避免双写和口径漂移；
3. 前端全站 UI 仍未完全稳定，新增摘要区块时要避免放大布局问题；
4. 结构化轨迹和网络摘要中的敏感数据需要持续强化脱敏；
5. 页面画像与摘要质量之间存在强相关，页面扫描质量不足时会影响摘要稳定性。

---

## 18. 不建议做的内容

当前阶段不建议：

1. 不做视频理解；
2. 不做 OCR；
3. 不引视觉模型；
4. 不引自主 Agent；
5. 不自动写正式知识库；
6. 不自动写正式 Mini-TOM；
7. 不让未确认摘要进入 Mini-TOM；
8. 不让未确认摘要进入 Weaviate / Neo4j；
9. 不追求第一版覆盖所有业务分支；
10. 不把摘要确认和 Mini-TOM 确认合并成一步。

---

## 19. 最终结论

“轨迹摘要文本化”应该作为 Mini-TOM 之前的必经中间层。

当前仓库已经足够支撑：

- 固定格式摘要草稿生成；
- 用户编辑 / 重生成 / 确认 / 驳回；
- 摘要门禁控制；
- 摘要作为后续测试用例生成和 Mini-TOM 的上游输入。

当前仓库还不足以支撑：

- 高置信自动正式沉淀；
- 未确认摘要直接参与正式知识资产；
- 不经人工确认的 Mini-TOM 正式写入。

因此，推荐路线是：

**结构化轨迹 → 轨迹摘要文本化草稿 → 人工确认 → Mini-TOM 候选 → 人工确认 → 正式沉淀**
