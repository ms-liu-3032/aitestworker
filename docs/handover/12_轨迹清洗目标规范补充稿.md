# 12 轨迹清洗目标规范补充稿

## 1. 目的

本稿用于沉淀当前已经打准、可复用的轨迹清洗模式，作为：

1. `TraceStepNormalizer` 继续迭代的目标基线；
2. 轨迹摘要文本化的稳定上游输入约束；
3. 测试用例草稿生成的业务动作参考；
4. “轨迹定位与语义修正建议”的候选对照标准。

本稿不覆盖所有业务，只覆盖当前已经验证通过的高频链路：

1. 取消邀请链；
2. 历史对象选择链；
3. 签到链；
4. 邀请变更链。

---

## 2. 总体清洗原则

### 2.1 要保留的内容

清洗后步骤应优先保留：

1. 关键页面与关键弹窗；
2. 关键输入字段与最终输入值；
3. 关键业务动作；
4. 关键提交、确认、取消动作；
5. 关键业务结果（如成功提示、跳转结果）。

### 2.2 要压缩或过滤的内容

应尽量压缩或过滤：

1. 输入法中间态；
2. 重复 click；
3. 页面心跳/辅助请求噪音（如 `GET /api/wx/getServerTime`）；
4. 成功后无意义的时间按钮点击；
5. 列表行对象、图标、辅助控件造成的歧义文本；
6. 同一业务动作的重复提交/重复确认。

### 2.3 业务语义优先级

清洗后的展示优先级应为：

1. 业务动作语义；
2. 字段级业务输入；
3. 页面/弹窗上下文；
4. 技术细节（接口、定位、selector）仅作为辅助，不应主导正文。

## 3. 取消邀请链

## 3.1 适用范围

适用于：

- 页面：`邀请记录`
- 后续动作：`取消邀请`
- 典型区块：
  - 取消原因输入框
  - 发送短信勾选项
  - 确认/取消按钮

## 3.2 目标语义

### 分支 A：取消邀请并发送短信

1. 打开邀请记录
2. 点击详情及操作
3. 发起取消邀请
4. 输入取消原因“最终中文值”
5. 选择“发送短信”
6. 确认取消邀请
7. 出现提示“操作成功”

### 分支 B：取消邀请且不发送短信

1. 打开邀请记录
2. 点击详情及操作
3. 发起取消邀请
4. 输入取消原因“最终中文值”
5. 保持“不发送短信”状态并确认取消邀请
6. 出现提示“操作成功”

### 分支 C：取消当前取消邀请操作

1. 打开邀请记录
2. 点击详情及操作
3. 发起取消邀请
4. 输入取消原因（可选）
5. 选择“发送短信”或“不发送短信”（可选）
6. 取消当前取消邀请操作

## 3.3 不应直接保留的原始表达

以下表达不应直接作为最终步骤保留：

1. `点击“关闭”`
2. `点击“确定”` / `点击“确 定”`
3. `勾选“确定”`
4. `qu'xiao / 取消yao'qing / 取消邀请ce'shi`
5. `点击 test001未开始test 按钮`
6. `点击 2026-05-19 15:45 按钮`

## 4. 历史对象选择链

## 4.1 适用范围

适用于：

- 页面：`新建邀请`
- 弹窗或区块：`添加对象`
- 子区域：`历史对象记录`

## 4.2 目标语义

### 选择历史对象

1. 打开添加对象弹窗
2. 选择历史对象“对象”
3. 继续选择其他历史对象（可选）
4. 提交添加对象
5. 出现提示“操作成功”

### 取消选择历史对象

1. 打开添加对象弹窗
2. 取消选择历史对象“对象”
3. 提交添加对象

## 4.3 目标对象表达

优先目标：

1. `姓名 / 手机`
2. `姓名`
3. `稳定主标识`

不应直接保留：

1. `0.87358400113361119468794544`
2. `86143****8908`
3. `test003 · test003`
4. `test13201503032 · 确定`

## 4.4 目标动作表达

优先表达为：

1. `选择历史对象“xxx”`
2. `取消选择历史对象“xxx”`

而不是：

1. `在“xxx”输入已勾选`
2. `点击“xxx · 确定”按钮`

---

## 5. 签到链

## 5.1 适用范围

适用于：

- 页面：`邀请记录`
- 操作列 / 详情及操作
- 接口：`POST /vsk/business-object/api/objects/print`
- 结果提示：`签到成功`

## 5.2 目标语义

1. 打开邀请记录
2. 点击详情及操作
3. 点击签到
4. 出现提示“签到成功”

## 5.3 允许的增强表达

如果行对象稳定，可增强为：

1. `对业务对象“xxx”执行签到`

如果对象不稳定，保留为：

1. `点击“签到”按钮`

## 5.4 不应直接保留的原始表达

不应再出现：

1. `点击“test003未开始test”按钮`
2. `点击“wafer”按钮`
3. 其他与 `objects/print` + `签到成功` 不匹配的歧义对象文本

## 6. 邀请变更链

## 6.1 适用范围

适用于：

- 页面：`邀请记录`
- 入口动作：`修改`
- 跳转页：`新建邀请`
- 通知确认弹窗：`是否需要向业务对象发送更新通知`

## 6.2 目标语义

### 分支 A：变更并发送更新通知

1. 打开邀请记录
2. 点击详情及操作
3. 发起邀请变更
4. 进入邀请变更页面
5. 修改对象信息或相关字段
6. 确认对象信息修改（如存在添加对象子弹窗）
7. 提交邀请变更
8. 选择“发送更新通知”
9. 跳转到邀请记录

### 分支 B：变更并不发送更新通知

1. 打开邀请记录
2. 点击详情及操作
3. 发起邀请变更
4. 进入邀请变更页面
5. 修改对象信息或相关字段
6. 确认对象信息修改（如存在添加对象子弹窗）
7. 提交邀请变更
8. 选择“不发送更新通知”
9. 跳转到邀请记录

### 分支 C：修改通知语言后提交

1. 打开邀请记录
2. 点击详情及操作
3. 发起邀请变更
4. 进入邀请变更页面
5. 选择通知语言“简体中文”
6. 提交邀请变更
7. 选择“发送更新通知”或“不发送更新通知”
8. 跳转到邀请记录

## 6.3 语义映射规则

当前已验证的关键映射包括：

1. `点击“修改”` + 跳转 `新建邀请`
   - `发起邀请变更`
2. `跳转到页面“新建邀请”`
   - `进入邀请变更页面`
3. 变更页中“添加对象”区块内 `点击“确定”`
   - `确认对象信息修改`
4. 变更页中的 `点击“提交”`
   - `提交邀请变更`
5. `点击“是”`
   - `选择“发送更新通知”`
6. `点击“否”`
   - `选择“不发送更新通知”`
7. `点击“简体中文”`
   - `选择通知语言“简体中文”`

## 6.4 不应直接保留的原始表达

1. `跳转到页面“新建邀请”`（应优先翻成变更语义）
2. `点击“是”按钮`
3. `点击“否”按钮`
4. 成功后无意义的时间按钮点击
5. 连续重复的 `提交邀请变更`

## 7. 与摘要文本化的关系

本稿定义的四类稳定模式，应直接作为：

1. 轨迹摘要文本化的高置信输入；
2. 测试用例草稿生成的主动作参考；
3. 轨迹定位与语义修正建议的对照标准。

### 7.1 可直接进入摘要正文的内容

以下内容可以高置信进入摘要正文：

1. 发起取消邀请 / 确认取消邀请
2. 选择“发送短信” / 保持“不发送短信”状态
3. 选择历史对象
4. 点击签到 / 签到成功
5. 发起邀请变更 / 提交邀请变更
6. 选择“发送更新通知” / 选择“不发送更新通知”

### 7.2 仍建议留在待确认事项的内容

以下内容在对象不稳定时，建议继续作为待确认事项：

1. 历史对象具体对象文本
2. 列表行对象精确名称
3. 复杂输入字段最终值（如证件号被输入法污染）

---

## 8. 当前验收结论

截至 2026-05-26，以下链路可视为 **P0 可用**：

1. 取消邀请链
2. 历史对象选择链
3. 签到链
4. 邀请变更链

P0 可用的含义是：

1. 已足够支撑轨迹摘要文本化；
2. 已足够支撑测试用例草稿生成；
3. 已足够作为“轨迹定位与语义修正建议”的第一批稳定基线；
4. 仍允许后续继续做对象级和字段级润色，但不再阻塞主流程。
