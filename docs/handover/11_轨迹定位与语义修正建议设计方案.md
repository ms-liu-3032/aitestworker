# 11 轨迹定位与语义修正建议设计方案

## 1. 背景与定位

当前平台已经具备：

1. 结构化轨迹采集；
2. 结构化轨迹清洗；
3. 轨迹摘要文本化；
4. 测试用例草稿生成；
5. Mini-TOM 候选抽取的后续演进方向。

但在真实业务轨迹中，仍然存在一类“不是采集失败，而是语义低置信”的问题，例如：

- 列表行对象识别不稳；
- checkbox / selection 语义不稳；
- 弹窗确认链路不稳；
- 输入法中间态污染最终输入值；
- 低置信业务动作映射容易误导摘要和用例草稿。

因此，当前阶段不应引入“自动回放自愈”或“脚本自愈”，而应增加一层：

**轨迹定位与语义修正建议**

该能力只作为**采集后的分析能力**，目标是：

1. 提升轨迹摘要文本化质量；
2. 提升测试用例草稿质量；
3. 为后续 Mini-TOM 候选抽取提供更干净的上游输入；
4. 让低置信内容有明确的候选承载层，而不是直接污染清洗结果正文。

同时，第一版不应停留在“诊断提示”层，而应支持：

1. 系统生成可读、可编辑的候选修正值；
2. 用户直接确认系统候选；
3. 用户手动改写候选后再确认；
4. 已确认结果即时作用于当前轨迹的清洗后步骤、摘要和用例草稿输入链。

当前真实使用反馈进一步说明，仅做“字段级候选修正”仍然不够。用户真正想修的，不只是单个词或单个值，而是：

1. 整条清洗后步骤文案；
2. 明显的噪音步骤；
3. 相邻重复步骤；
4. 多次确认后可复用的相同模式。

因此，本方案从当前版本开始，明确升级为：

> **字段级修正 + 步骤级修正 + 模式学习辅助**

### 1.1 正确定位

本能力是：

- 候选分析层；
- 可编辑修正层；
- 人工确认前置层；
- 结构化轨迹清洗与轨迹摘要文本化之间的过渡层。

本能力不是：

- 自愈引擎；
- 自动回放修复；
- 自动脚本修复；
- 自动 locator 替换；
- 自动正式资产写入器。

### 1.2 能力链路位置

推荐链路：

**结构化轨迹**
→ **基础清洗**
→ **轨迹定位与语义修正建议**
→ 用户确认 / 驳回
→ **轨迹摘要文本化**
→ 用户确认 / 驳回
→ **测试用例草稿生成**
→ 后续 Mini-TOM 候选抽取

### 1.3 当前阶段边界

当前阶段明确**不做**：

1. 不自动回放页面；
2. 不自动继续点击；
3. 不自动修复 Playwright 脚本；
4. 不自动替换正式 locator；
5. 不自动写入 Mini-TOM；
6. 不自动写入 Weaviate；
7. 不自动写入 Neo4j；
8. 不自动影响其他用户；
9. 不引入自主 Agent。

### 1.4 当前阶段为什么不是“自动自愈”

当前阶段要解决的问题，本质上是：

1. 语义低置信；
2. 操作步骤表述不准确；
3. 输入法中间态、重复动作与焦点噪音污染摘要和用例草稿。

这些问题适合通过：

- 候选修正；
- 人工改写确认；
- 当前轨迹即时生效；
- 已确认结果驱动后续候选质量提升

来处理，而不是通过页面回放、自动点击、脚本级 locator 替换来处理。

因此当前阶段只借“自愈”的思路做：

> **轨迹定位与语义修正建议**

而不做真正的回放自愈或执行自愈。

- `TraceStepNormalizer`
- `TraceAssetService`
- `TraceSummaryService`
- `TraceDataService`
- `BrowserTraceSessionService`
- `LlmGateway`
- `ControlledScanService`

### 2.4 当前缺口

当前仓库尚未形成一层正式的“修正建议”承载结构：

1. 没有专门的修正建议表；
2. 没有修正建议状态流转；
3. 没有前端“待确认建议”区块；
4. 摘要生成无法显式消费“已确认修正建议”；
5. 低置信内容只能硬塞进清洗结果，或直接丢失。

### 2.5 现状结论

当前仓库已经具备实现“轨迹定位与语义修正建议”所需的主要底座，但还缺：

> **候选承载层 + 确认工作流 + 摘要消费门禁。**

## 3. 第一版范围定义

第一版只处理以下五类问题。

### 3.1 列表行对象识别候选

示例问题：

- `test003未开始test`
- `86143****8908`
- `0.87358400113361119468794544`

目标：

- 给出该行主对象的候选值；
- 指出为什么当前识别低置信；
- 允许用户确认或驳回。

### 3.2 checkbox / selection 语义候选

示例问题：

- `已勾选`
- `已取消勾选`
- 默认未勾选但直接确认

目标：

- 候选翻译为：
  - `选择“发送短信”`
  - `选择“不发送短信”`
  - `保持“不发送短信”状态`

### 3.3 弹窗确认链路候选

示例问题：

- `关闭`
- `确定`
- `确 定`
- 连续多个 click

目标：

- 候选翻译为：
  - `发起取消邀请`
  - `确认取消邀请`
  - `取消当前取消邀请操作`

### 3.4 输入法中间态合并与最终值候选

示例问题：

- `qu'xiao`
- `取消yao'qing`
- `取消邀请ce'shi`
- `取消邀请测试`

目标：

- 提供最终值候选，例如：
  - `取消邀请测试`
- 并记录中间态污染来源。

### 3.5 低置信业务动作映射待确认提示

示例问题：

- `点击“wafer”按钮`
- `点击“test”按钮`
- `点击“关闭”按钮`

目标：

- 提示它们可能对应的业务动作；
- 不把这些低置信解释直接当成事实。

### 3.6 步骤级修正（新增）

这是本次设计修正后的关键补充。

仅靠字段级候选修正，无法覆盖以下真实需求：

1. 把 `点击“详情及操作”按钮` 改写成 `点击第 1 条记录的“详情及操作”按钮`；
2. 把 `点击“关闭”按钮` 改写成 `发起取消邀请`；
3. 把多条输入法中间态输入压成一条最终输入；
4. 删除成功后时间按钮、焦点按钮等噪音步骤；
5. 把相邻重复 click 或重复提交动作合并为一条业务动作。

因此第一版应新增“步骤级修正”能力，至少支持：

- `STEP_TEXT_OVERRIDE`：改写整条步骤文案；
- `STEP_NOISE_DECISION`：删除噪音步骤；
- `STEP_MERGE_SUGGESTION`：建议与前后步骤合并。

目标不是自动改正式事实，而是：

> 为当前轨迹生成可编辑、可确认的步骤级修正候选。

### 3.7 模式学习辅助（新增）

当前真实诉求还包括：

> “记录学习多次修正之后，后续生成的操作记录能更准确”

本阶段不做黑箱学习或自主学习，而是做：

**已确认修正结果驱动的轻量模式复用**

可沉淀的模式包括：

1. 同页面相同按钮的业务语义映射；
2. 相同控件文本的对象识别修正；
3. 相同 checkbox / selection 语义修正；
4. 相同输入法中间态的最终值修正方式；
5. 相同重复 click / 中间态步骤的噪音判定方式。

模式学习的目标不是自动影响所有轨迹，而是：

- 提升后续候选建议质量；
- 降低再次人工改写成本；
- 仍然保留人工确认门禁。

## 4. 状态模型

本能力所有结果都必须是候选。

### 4.1 状态流转

只允许三条路径：

1. `CANDIDATE -> CONFIRMED`
2. `CANDIDATE -> 手动改写 -> CONFIRMED`
3. `CANDIDATE -> REJECTED`

### 4.2 状态语义

- `CANDIDATE`
  - 系统给出的候选建议；
  - 不能视为确定事实；
  - 只能进入“待确认事项”。

- `CONFIRMED`
  - 用户确认后的修正结果；
  - 可以是“直接确认系统候选”，也可以是“人工改写后确认”；
  - 可以用于当前摘要、用例草稿、Mini-TOM 候选上游输入。

- `REJECTED`
  - 用户明确驳回；
  - 必须保留驳回原因；
  - 后续生成不得再默认采用。

### 4.3 修正粒度

从本次设计修正开始，修正建议分为两层粒度：

#### A. 字段级修正

适用于：

- 对象文本；
- checkbox 语义；
- 输入最终值；
- 低置信业务动作映射。

#### B. 步骤级修正

适用于：

- 改写整条 clean step 文案；
- 删除整条噪音步骤；
- 合并相邻重复步骤；
- 调整同一业务动作的步骤级表达。

字段级修正主要修“词”，步骤级修正主要修“句”和“步骤序列”。

当前真实业务反馈表明：

> 当前阶段已经不能只修“词”，必须开始支持修“步骤”。

---

## 5. 确认后的作用范围

### 5.1 `CONFIRMED` 的作用范围

已确认修正结果可以用于：

1. 当前轨迹摘要文本化；
2. 当前测试用例草稿生成；
3. 后续 Mini-TOM 候选抽取的上游输入。

### 5.2 `CANDIDATE` 未确认结果的作用范围

未确认结果只能用于：

- 待确认事项展示；
- 候选修正预览；
- 人工审阅。

不得：

- 作为摘要正文里的确定事实；
- 进入 Mini-TOM；
- 进入 Weaviate；
- 进入 Neo4j；
- 影响其他用户默认生成。

## 6. 数据结构建议

建议新增正式承载层：

`trace_correction_candidate`

不建议命名为：

- `self_heal`
- `healing`
- `auto_fix`
- `recovery`

### 6.1 建议字段

#### 关联字段

- `id`
- `project_id`
- `trace_group_id`
- `trace_session_id`，可空
- `issue_clip_id`，可空
- `trace_event_id`，可空
- `summary_id`，可空

#### 候选本体

- `correction_type`
  - `OBJECT_LABEL`
  - `CHECKBOX_SEMANTICS`
  - `DIALOG_ACTION`
  - `INPUT_FINAL_VALUE`
  - `BUSINESS_ACTION_MAPPING`
- `source_text`
- `candidate_value`
- `confirmed_value`，可空
- `candidate_reason`
- `confidence_label`
  - `HIGH`
  - `MEDIUM`
  - `LOW`

#### 状态

- `status`
  - `CANDIDATE`
  - `CONFIRMED`
  - `REJECTED`

#### 快照与溯源

- `raw_context_snapshot`
- `prompt_snapshot_id`
- `context_manifest_id`
- `llm_invocation_log_id`

#### 审核字段

- `created_by`
- `confirmed_by`
- `confirmed_at`
- `rejected_by`
- `rejected_at`
- `rejected_reason`
- `created_at`
- `updated_at`

#### 步骤级修正补充字段

为了支持“修步骤而不是只修词”，建议新增以下字段，或在当前结构中补充等价表达：

- `step_no`
- `correction_scope`
  - `VALUE`
  - `STEP`
  - `PATTERN`
- `operation_type`
  - `REWRITE`
  - `DROP`
  - `MERGE`
- `candidate_step_text`
- `confirmed_step_text`
- `related_step_no`

### 6.2 设计原则

1. 不与 `trace_summary` 混表；
2. 不直接复用 `knowledge_asset`；
3. 不把“建议”伪装成“正式摘要”；
4. 每条建议必须能追溯到原轨迹上下文和生成快照；
5. `candidate_value` 是系统猜测，`confirmed_value` 才是最终采用值；
6. 未确认建议不得覆盖当前轨迹事实。
7. 若已支持字段级修正但尚未支持步骤级修正，应优先通过新增最小字段补齐步骤级修正，不要继续用 `candidate_value` 硬扛整句改写。

## 7. 服务设计

建议新增：

`TraceCorrectionSuggestionService`

### 7.1 核心方法

#### 1. `generateSuggestions(traceGroupId, scope)`

作用：

- 基于单个轨迹范围生成修正建议候选。

输入：

- `trace_group_id`
- `summary_scope`
- `trace_session_id?`
- `issue_clip_id?`
- `model_config_id`

输出：

- `trace_correction_candidate[]`

要求：

- 默认状态为 `CANDIDATE`
- 必须落：
  - `raw_context_snapshot`
  - `prompt_snapshot_id`
  - `context_manifest_id`
  - `llm_invocation_log_id`

#### 2. `listSuggestions(traceGroupId)`

作用：

- 按 group / scope 查看修正建议列表。

#### 3. `confirmSuggestion(id, confirmedValue?)`

作用：

- 将建议状态切为 `CONFIRMED`
- 写入 `confirmed_by / confirmed_at`

规则：

- 如未传 `confirmedValue`，则按 `candidate_value` 直接确认；
- 如传入 `confirmedValue`，则视为“手动改写后确认”；
- 最终采用值统一写入 `confirmed_value`。

#### 4. `rejectSuggestion(id, reason)`

作用：

- 将建议状态切为 `REJECTED`
- 写入 `rejected_by / rejected_at / rejected_reason`

#### 5. `applyConfirmedSuggestionsToSummary(summaryId)`

作用：

- 将已确认建议应用到指定摘要生成输入链；
- 只在摘要生成或摘要重生成时消费；
- 不直接回写原始轨迹事件。

#### 6. `applyConfirmedSuggestionsToCleanSteps(traceGroupId, scope)`

作用：

- 将步骤级和字段级的 `CONFIRMED` 修正同时应用到当前轨迹 clean steps；
- 支持：
  - 文案改写
  - 噪音删除
  - 步骤合并

#### 7. `learnPatternFromConfirmedSuggestions(traceGroupId)`

作用：

- 基于当前轨迹中已确认修正生成轻量模式记忆；
- 仅用于提升后续候选建议质量；
- 不自动回写正式资产。

### 7.2 设计边界

该服务不允许：

1. 回放页面；
2. 重做点击；
3. 修改 Playwright 脚本；
4. 覆盖原始 locator；
5. 自动写正式资产。

并且当前阶段不允许：

6. 直接把模式学习结果当成确定事实覆盖新轨迹；
7. 绕过人工确认门禁直接改写 clean steps。

## 8. API 设计

建议新增以下接口：

### 8.1 生成建议

`POST /api/trace/groups/{groupId}/corrections:generate`

作用：

- 基于指定轨迹范围生成修正建议候选。

### 8.2 获取建议列表

`GET /api/trace/groups/{groupId}/corrections`

作用：

- 查看该轨迹组下的所有修正建议。

### 8.3 确认建议

`POST /api/trace/corrections/{id}:confirm`

作用：

- 将候选修正标记为已确认。

### 8.4 驳回建议

`POST /api/trace/corrections/{id}:reject`

作用：

- 驳回某条候选修正，并记录原因。

### 8.5 可选接口

`POST /api/trace/summaries/{summaryId}:apply-corrections`

作用：

- 在摘要重生成前，将已确认修正建议注入摘要输入链。

`POST /api/trace/groups/{groupId}/step-corrections:apply`

作用：

- 将已确认的步骤级修正即时应用到当前轨迹 clean steps 展示；
- 仅作用于当前轨迹上下文；
- 不写正式资产。

## 9. 前端页面设计

建议在轨迹详情页新增正式区块：

## 轨迹定位与语义修正建议

### 9.1 展示分组

建议按以下分组展示：

1. 对象识别建议
2. checkbox / selection 语义建议
3. 弹窗确认链路建议
4. 输入最终值建议
5. 低置信业务动作待确认提示
6. 操作记录修正建议

### 9.2 每条建议展示内容

每条建议至少展示：

- 原始文本
- 候选值
- 已确认值（如存在）
- 候选原因
- 置信度
- 当前状态

若为步骤级修正，还应展示：

- 原始步骤文案
- 候选步骤文案
- 修正动作类型（改写 / 删除 / 合并）

### 9.3 用户操作

每条建议至少支持：

- `确认`
- `手动改写后确认`
- `驳回`

若为步骤级修正，还应支持：

- 改写整条步骤文案
- 删除该步骤
- 标记与前后步骤合并

### 9.4 展示原则

不要做成：

- 自动修复中心
- 回放修复中心
- 自动执行入口

只做：

- 候选建议
- 待确认事项
- 确认 / 驳回工作流

并且：

1. 对于低质量候选，不应只展示泛泛提示文案；
2. 应优先给出一个可编辑的候选值；
3. 用户确认后应立即刷新当前轨迹的清洗后步骤与摘要输入预览。
4. 对于用户明确希望修正的 clean step，应优先提供“步骤级改写 / 删除 / 合并”的交互。

## 10. 与轨迹摘要文本化的关系

### 10.1 新链路

旧链路：

结构化轨迹
→ 直接摘要文本化

新链路：

结构化轨迹
→ 基础清洗
→ 轨迹定位与语义修正建议
→ 用户确认 / 驳回
→ 轨迹摘要文本化

### 10.2 摘要消费规则

1. `CONFIRMED` 修正建议可以参与摘要生成；
2. `CANDIDATE` 未确认建议只能显示在摘要的“待确认事项”中；
3. 未确认建议不得作为摘要正文中的确定事实；
4. 摘要正文应优先使用已确认的修正结果。

若为步骤级修正，则：

5. 已确认的步骤改写应优先影响摘要正文里的关键步骤表述；
6. 已确认的噪音删除不应继续进入摘要正文；
7. 已确认的步骤合并应先体现在 clean steps，再体现在摘要输入。

### 10.3 当前轨迹即时应用原则

在当前轨迹详情页中：

1. `CONFIRMED` 修正建议应即时影响“清洗后步骤”展示；
2. `CONFIRMED` 修正建议应即时影响当前轨迹摘要重生成输入；
3. `CONFIRMED` 修正建议应即时影响当前测试用例草稿生成输入；
4. 该即时应用仅作用于当前轨迹上下文，不得影响其他用户、其他轨迹组或正式资产。

## 11. 与测试用例生成、Mini-TOM 的门禁关系

### 11.1 对测试用例草稿

1. `CONFIRMED` 修正建议可以参与当前测试用例草稿生成；
2. `CANDIDATE` 未确认建议只能作为低置信参考；
3. 未确认建议生成的测试用例仍只允许进入用户本地用例库；
4. 不得自动进入正式用例库。

若为步骤级修正，则：

5. 已确认的步骤改写应直接进入用例草稿步骤正文；
6. 已确认的噪音删除应避免继续污染用例步骤；
7. 已确认的步骤合并应减少重复和流水账。

### 11.2 对 Mini-TOM

必须满足：

1. 只有 `CONFIRMED` 的修正建议，才允许进入后续 Mini-TOM 候选抽取；
2. 只有 `CONFIRMED` 的 `trace_summary`，才允许进入 Mini-TOM 候选抽取；
3. `CANDIDATE / REJECTED` 修正建议不得进入 Mini-TOM；
4. 未确认 Mini-TOM 不得进入正式资产。

模式学习结果也不得绕过这条门禁：

- 可用已确认修正提升后续候选质量；
- 但不能把“模式记忆”直接当成正式事实写入 Mini-TOM。

### 11.3 对 Weaviate / Neo4j

必须满足：

1. 未确认修正建议不得进入 Weaviate；
2. 未确认修正建议不得进入 Neo4j；
3. 未确认修正建议不得影响其他用户默认生成。

---

## 12. 第一版验收标准

第一版验收只看以下内容：

1. 能基于单个 `trace_group` 生成修正建议；
2. 能生成对象识别候选；
3. 能生成 checkbox 语义候选；
4. 能生成弹窗确认链路候选；
5. 能生成输入最终值候选；
6. 能生成整条操作记录的步骤级改写候选；
7. 能生成明显噪音步骤的删除候选；
8. 每条建议默认状态为 `CANDIDATE`；
9. 用户可以确认建议；
10. 用户可以手动改写候选后确认；
11. 用户可以驳回建议；
12. 已确认建议能参与摘要生成；
13. 已确认建议能参与测试用例草稿生成；
14. 已确认建议能即时影响当前轨迹的清洗后步骤展示；
15. 未确认建议只能进入待确认事项；
16. 未确认建议不能进入 Mini-TOM；
17. 未确认建议不能写入 Weaviate / Neo4j；
15. 不做自动回放，不做自动点击，不做脚本自愈。

---

## 13. 风险点

### 13.1 结构化语义底层仍不完全稳定

如果原始轨迹：

- 对象识别噪音过大；
- checkbox 语义完全缺失；
- 输入法中间态污染严重；

则候选修正建议质量也会下降。

### 13.2 建议层与摘要层边界容易被做混

如果不严格区分：

- `CANDIDATE`
- `CONFIRMED`

就容易把候选解释误当成摘要事实。

同样，如果只支持“确认 / 驳回”而不支持“手动改写后确认”，就会让修正建议退化成“诊断提示”，无法真正提升当前轨迹清洗结果质量。

### 13.3 用户确认负担

如果建议粒度过碎，用户会觉得累。  
第一版必须控制候选数量，优先只暴露最影响摘要和用例草稿质量的建议。

---

## 14. 不建议做的内容

当前阶段不建议做：

1. 不叫“自愈引擎”；
2. 不做自动回放修复；
3. 不做自动 locator 替换；
4. 不做脚本级重试与修复；
5. 不把修正建议直接写成正式 Mini-TOM；
6. 不把修正建议直接写进知识图谱；
7. 不把修正建议做成跨用户共享默认事实；
8. 不把“只有确认/驳回、不可编辑”的弱提示误当成完整方案；
9. 不引入自主 Agent。

---

## 15. 结论

“轨迹定位与语义修正建议”的正确定位是：

> **借“自愈”的思路做候选修正，但绝不做自动回放、自动修脚本、自动正式沉淀。**

同时，这层能力不应只停留在“建议用户自己理解”的提示层，而应进一步支持：

> **候选修正值生成 -> 人工可编辑 -> 确认后立即作用于当前轨迹**

并且从当前真实需求出发，必须进一步明确：

> **“轨迹定位与语义修正建议”不能只修字段，还要开始支持修步骤。**

它的价值在于：

1. 给低置信内容一个合规落点；
2. 减轻基础清洗器一次性拍板的压力；
3. 提高轨迹摘要文本化质量；
4. 提高测试用例草稿质量；
5. 为后续 Mini-TOM 候选抽取提供更稳定上游输入。

因此，当前阶段最合适的路径不是“自动自愈”，而是：

**基础清洗**
→ **轨迹定位与语义修正建议**
→ **人工确认**
→ **摘要 / 用例 / Mini-TOM 上游输入**

进一步说，当前阶段最合适的实现方向不是：

- 自动自愈
- 自动回放
- 自动执行修复

而是：

> **字段级候选修正 + 步骤级候选修正 + 已确认结果驱动的模式学习辅助**
