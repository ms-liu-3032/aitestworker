## 6. 多人共用 LLM Token 数据污染治理专项

详细方案见 [07_LLM数据污染治理方案.md](07_LLM数据污染治理方案.md)。本节只给入口与排期。

### 背景

多人共用同一个 LLM API Token 不是数据污染的根因，污染来自上下文拼装、RAG 检索、Skill / Tool / Prompt、草稿资产、历史用例、轨迹资产、缓存与日志没做隔离。本专项不要求"每人一个 Token"，但要求平台侧做严格的隔离、过滤、版本快照与审计。

### 核心抓手

1. 新建后端唯一入口 `LlmGateway`，禁止前端直连模型、禁止业务服务直接 `LlmAdapter`。
2. 每次调用必落 `prompt_snapshot` + `context_manifest` + `llm_invocation_log`。
3. RAG 检索必须带 `projectId / scope / status / visibility` 过滤，弃用资产默认排除。
4. 用户本地用例库（`test_case_draft / PERSONAL`）与正式用例库（`test_case_asset / PROJECT`）严格分离，AI 生成永不直接进正式库与向量库。
5. Skill / Tool / Prompt 全部加 scope / status / review_status / version；普通用户改公共提示词只生成本次 snapshot，不动模板原文。
6. Prompt Injection 防护层 + 安全事件日志。
7. 后台新增 AI 调用审计页面与安全事件页面。

### 排期（与本节一同管理，对齐 §3 当前优先级）

| 循环 | 工时 | 内容 |
|---|---|---|
| 循环 1 — 骨架 | ~1 周 | V7 Flyway、LlmGateway 主流程、第 1 个调用点接入、调用日志开写 |
| 循环 2 — 上下文与权限 | ~1 周 | RAG 过滤、Context Manifest、Prompt Snapshot、弃用资产排除、缓存 key 守卫 |
| 循环 3 — 用例库分层 | ~1 周 | PERSONAL/PROJECT 切分、提交/导出流程、前端两 Tab、模型直连守卫 |
| 循环 4 — 资产治理 | ~1.5 周 | skill_template / tool_template、prompt 版本与 fork、trust_level |
| 循环 5 — 安全与审计 | ~1.5 周 | PromptInjectionGuard、security_event_log、审计页面、trace_summary 与标签 |
| 循环 6 — 限流与多租户 | 视需求 | LLM 限流/预算、多模型路由、多租户 API Key |

每个循环独立可上线、可回退；不允许把 P0 的多个子任务合到一个 PR。

### 任务标记

- P0：循环 1–3 的全部条目（详见 07 文档 §23.1）
- P1：循环 4–5 的全部条目（07 文档 §23.2）
- P2：循环 6 的全部条目（07 文档 §23.3）

### 验收口径

- 任何 LLM 调用都能在审计页查到，含 user/project/task/stage/model/prompt 版本/included assets。
- 用户 A 的 PERSONAL 草稿永远不出现在用户 B 的检索结果。
- 弃用资产默认不在任何检索结果中。
- 历史 task 即使在 prompt_template 改版之后，依然可以基于 prompt_snapshot 复现。
- 共用同一 model_config + 同一 Token 的多用户并发调用，互不串味（单元测试覆盖）。
