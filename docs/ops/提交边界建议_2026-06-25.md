## 如果想压成 2 个提交

也可以压成下面两次：

### 方案 A
1. `chore: clean open-source repository boundary and docs`
2. `feat: stabilize generation, trace, and worker workflows`

适合：
- 你更重视快速收口
- 不在意 review 时前后端/worker 混在一个功能提交里

---

## 不建议混在一起的内容

下面这些最好不要和别的提交混成一个“大杂烩”：

1. 文档删除 / 开源清洗  
   原因：这是仓库边界收口，不应和功能修复绑死

2. Flyway / 增量分析迁移修复  
   原因：这是数据库契约变更，最好和生成链路一起提交

3. worker 纳入仓库  
   原因：它会让 diff 形态明显变化，独立一笔更清楚

---

## 建议提交顺序

```text
Commit 1 -> Commit 2 -> Commit 3
```

理由：
- 先把仓库边界收干净
- 再提交核心主链路修复
- 最后补 trace / worker / UI 收尾
