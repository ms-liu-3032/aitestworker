# V16 Flyway 内测环境恢复说明

## 适用场景
适用于已部署过旧版 `V16__concurrency_governance.sql`，并在启动时出现以下任一症状的内测环境：

- `Duplicate key name 'idx_tom_source'`
- `Duplicate column name 'version'`
- `Migration checksum mismatch for migration version 16`
- `flyway_schema_history` 中存在 `version = 16, success = 0`

## 根因概述
`V16__concurrency_governance.sql` 在部分环境中发生过“半执行成功后失败”：

1. 一部分 `version` 列已经落库
2. 一部分索引已经存在
3. 随后迁移失败，Flyway 记录为 `success = 0`
4. 历史迁移文件内容之后又发生变化，触发 checksum mismatch

因此当前故障不是单纯重启容器能解决的问题，而是 Flyway 历史迁移事故。

## 上线前提
在执行恢复前，必须先确认待部署 backend 包中已包含当前仓库版本的：

- `backend/src/main/resources/db/migration/V16__concurrency_governance.sql`

不要在旧 backend 包上直接做数据库修复后重试。

## 建议恢复步骤

### 1. 备份数据库
```bash
docker compose exec mysql sh -c 'mysqldump -uroot -proot aitest > /tmp/aitest-before-v16-retry.sql'
docker compose cp mysql:/tmp/aitest-before-v16-retry.sql ./aitest-before-v16-retry.sql
```

按实际环境账号密码替换 `root/root`。

### 2. 检查 Flyway 失败记录
```bash
docker compose exec mysql mysql -uroot -proot -D aitest -e "
SELECT installed_rank, version, description, checksum, success
FROM flyway_schema_history
WHERE version = '16'
ORDER BY installed_rank;
"
```

如果看到：
- `version = 16`
- `success = 0`

说明当前环境处于 V16 半执行失败状态。

### 3. 检查关键残留对象
```bash
docker compose exec mysql mysql -uroot -proot -D aitest -e "
SHOW COLUMNS FROM test_object_model LIKE 'version';
SHOW COLUMNS FROM trace_summary LIKE 'version';
SHOW COLUMNS FROM test_case_draft LIKE 'version';
SHOW COLUMNS FROM test_point_draft LIKE 'version';
SHOW COLUMNS FROM test_case_asset LIKE 'version';
SHOW INDEX FROM test_object_model WHERE Key_name = 'idx_tom_source';
SHOW INDEX FROM test_object_model WHERE Key_name = 'idx_tom_scope_status';
"
```

如果这些列/索引已存在，不要手工删除；新版 V16 会在对象已存在时跳过。

### 4. 删除失败的 V16 Flyway 记录
```bash
docker compose exec mysql mysql -uroot -proot -D aitest -e "
DELETE FROM flyway_schema_history
WHERE version = '16' AND success = 0;
"
```

### 5. 部署新版 backend 包后重启服务
```bash
docker compose up -d backend web
docker compose logs -f backend
```

## 预期结果
backend 日志应表现为：

1. schema 当前版本从 `15` 开始继续迁移
2. `V16` 在已存在列/索引上安全跳过
3. 继续执行后续迁移
4. Spring Boot 正常启动

## 不建议的操作
- 不要直接把 `success = 0` 改成 `success = 1`
- 不要手工删除 `version` 列
- 不要手工删除 `idx_tom_source`
- 不要在旧 backend 包上反复重启碰运气

## 后续要求
这次事故说明历史迁移文件一旦进入环境，不应再随意改写而不配套恢复说明。后续如果再调整 V16 相关策略，必须同步更新这份恢复文档和发布说明。
