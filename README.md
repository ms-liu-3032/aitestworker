# 受控智能 AI 测试平台

一个基于 Playwright 轨迹采集、Mini-TOM 测试对象模型、LLM Wiki 知识包、项目语义上下文和大语言模型的智能测试平台。平台把真实操作、页面结构、业务规则和测试经验沉淀为可追溯资产，并通过需求范围审核、测试点拆解、测试点范围审核和用例编排，持续生成可审计、可维护的测试用例草稿。

平台默认启动时为纯通用内核，不绑定任何特定业务。业务能力通过项目使用过程中自动沉淀的 business pack 持续演化。

## 快速开始

### Docker Compose（推荐）

```bash
cd packaging
./build-server-package.sh
cd ../release/server/<最新包目录>
./install.sh
```

启动后访问：

- 前端：`http://localhost:35173`
- 后端：`http://localhost:38080`

### Docker 日志与排障

服务端默认以 Docker Compose 方式部署，运行日志通过容器标准输出查看，不依赖宿主机绝对路径。

先查看容器状态：

```bash
docker compose ps
```

查看后端日志：

```bash
docker logs aitest-server-backend --tail 200
```

查看前端 Nginx 日志：

```bash
docker logs aitest-server-web --tail 100
```

查看 MySQL 日志：

```bash
docker logs aitest-server-mysql --tail 100
```

如果部署包中的 `.env` 修改了 `APP_NAME`，则容器名会变为 `${APP_NAME}-backend`、`${APP_NAME}-web`、`${APP_NAME}-mysql`。

### CORS 白名单配置

默认配置兼容本地开发和小范围内测：

```bash
APP_CORS_ALLOWED_ORIGINS=*
```

正式公网部署建议改为实际前端访问地址，支持逗号分隔多个 Origin。Origin 需要包含协议、IP/域名和端口：

```bash
APP_CORS_ALLOWED_ORIGINS=http://服务器IP,http://服务器IP:35173,https://example.com
```

如果前后端通过同一个 Nginx 地址访问，通常不会触发跨域；如果前端和后端不同端口，则把浏览器访问的前端地址加入白名单即可。只写 IP（例如 `1.2.3.4`）不是合法 Origin，应写成 `http://1.2.3.4` 或 `http://1.2.3.4:端口`。

## 核心能力

### 轨迹采集与清洗
- 执行轨迹采集器：身份空间、采集组、Chrome 打开、事件采集、网络摘要
- 通用轨迹清洗引擎：基于 JSON 规则包的输入/变更/点击描述规范化
- 工作流处理器：对话输入确认流、入口导航提交流等可扩展流程
- 轨迹规则包：默认不加载业务样例，支持全局 / 项目级 `trace_rule_pack_config` 数据库配置，classpath JSON 仅作为兼容模式

### 语义沉淀
- 项目语义上下文：从 TOM、页面画像、步骤模板、业务摘要自动构建
- 语义包快照：`semantic_pack` / `semantic_pack_item` 自动刷新
- 业务包（Business Pack）：从项目资产自动生成，支持：
  - 自动沉淀：项目使用过程中持续演化
  - 生命周期：DRAFT → ACTIVE → INACTIVE → ARCHIVED
  - 来源追溯：每个条目可追溯到 TOM、页面画像、轨迹摘要等项目资产
  - 绑定关系：与规则引擎、扫描、TOM、语义包正式关联
  - 版本快照：每次状态变更自动记录快照
  - 包间关系：支持依赖、包含、联动、补充关系
  - 消费记录：追踪哪些链路消费了业务包
  - 自动刷新诊断：记录自动沉淀成功/失败、输入资产规模、生成包和推断关系数量
  - 可视化：前端可查看来源构成、来源跳转、绑定覆盖、消费链路、刷新诊断和关系图谱

### 测试对象模型（TOM）
- 候选抽取：从轨迹摘要、使用手册、测试资产自动抽取
- 人工确认：候选 → 生效 → 弃用 → 恢复完整生命周期
- 测试范围分析：基于 TOM 的需求影响分析，支持 LLM 语义匹配
- 项目级 / 系统级：支持 TOM 升级为系统级共享资产
- 三种使用模式：不使用 TOM、只使用项目 TOM、同时使用项目与系统 TOM；模式会贯穿分析、测试点和用例生成任务

### 用例生成
- 对话式生成工作台：需求输入 → 需求范围审核 → 测试点拆解 → 测试点范围审核 → 用例编排 → 草稿生成
- 双范围门禁：需求原子和测试点均可设为“生成 / 仅参考 / 排除”，未确认时不能绕过审核直接生成
- 审计保留：仅参考和排除项仍保留原始证据、AI 建议和人工决定，但排除项不会进入后续模型输入
- 大规模审核：测试点支持搜索、分页、当前筛选批量处理和全量批量处理
- 草稿治理：编辑、复制、确认、舍弃、批量操作、提交正式库完整链路
- 用例库性能：本地库和正式库采用服务端筛选分页，列表加载轻量摘要，详情按需读取
- 来源追溯：需求生成 / 轨迹回放双来源可见
- 异步生成任务：长耗时生成返回 `taskId`，前端轮询任务状态，避免 HTTP 请求长时间阻塞
- 重复点击保护：按项目、用户、任务类型、输入内容、提示词版本和模型配置生成请求指纹，相同运行中任务直接复用
- 草稿保护：模型生成结果只进入草稿区，正式用例库仍需人工确认

### LLM 调用治理
- 统一网关：业务代码统一通过 `LlmGateway` 调用模型，便于后续扩展 OpenAI 兼容网关、CCS 或其他提供商
- 超时与重试：支持连接超时、读取超时、任务级超时；仅对超时、限流、供应商异常等可恢复错误做有限重试
- 错误分类：输出机器可读错误码和用户可读提示，覆盖 `TIMEOUT`、`RATE_LIMITED`、`PROVIDER_ERROR`、`AUTH_ERROR`、`INSUFFICIENT_BALANCE`、`CONTEXT_TOO_LONG`、`OUTPUT_PARSE_ERROR`、`INVALID_REQUEST`、`UNKNOWN_ERROR`
- JSON 清洗：自动处理 markdown code fence、前后说明文本和 JSON 对象/数组提取，解析失败不会写入正式资产
- 调用日志：记录模型、耗时、重试次数、错误码和安全截断后的输出摘要，不记录 API Key 等敏感凭据
- 部署与排障细节见 [LLM 调用治理与异步生成说明](docs/ops/LLM调用治理与异步生成说明.md)

### 需求分析与用例生成主链路

平台在"对话式用例生成工作台"中实现了完整的分析-澄清-生成闭环：

**流程**：需求输入 → 需求理解与风险扫描 → 需要澄清 → 用户补充 → 需求范围确认 → 覆盖矩阵与测试点拆解 → 测试点范围确认 → 用例编排 → 生成草稿

**核心能力**：
- **需求理解**：自动识别需求类型（规则/表单/UI/状态/数据/混合），理解业务目标和影响范围
- **风险扫描**：评审前自动识别异常场景、边界条件、审批前置问题，问题带来源依据
- **澄清问题前置**：分析阶段发现信息不足时主动反问，用户补充后自动重新分析，直到信息充分才进入生成
- **需求范围确认**：AI 识别本期需求、背景资料和非本期内容，用户确认后才生成覆盖矩阵与测试点
- **类型化测试点拆解**：按需求类型选拆解骨架（规则型→条件组合/边界值；表单型→字段校验/必填；UI 型→交互流程/状态切换；状态型→流转/异常恢复；数据型→一致性/并发/幂等）
- **测试点范围确认**：测试点生成后由用户决定哪些进入用例编排，避免背景说明和非本期内容造成用例外溢
- **分阶段异步执行**：需求理解、矩阵/测试点、用例编排和草稿生成分别执行并保存检查点，失败可从未完成节点继续
- **输入源识别**：区分 PRD 文本、蓝湖/原型设计稿、TOM 模型等来源，蓝湖/原型类输入自动追加风险提示
- **证据约束**：TOM、页面画像、业务包、轨迹摘要作为证据来源，无证据的内容标记 `LOW_EVIDENCE` 和 `needs_confirmation`，不允许伪装成已确认事实
- **低证据保护**：低置信度结果不会冒充高置信完整结果，测试点明确标注覆盖状态和待确认项

**状态机**：
- 需求范围和测试点范围均确认、用例编排完成后，用户输入"生成用例"才进入草稿生成阶段
- 用户补充后自动重新分析，直到信息充分
- "重新分析"可显式触发全量重新分析

### 页面扫描
- 配置化扫描源：支持全局 / 项目级扫描源配置；项目页只管理项目级扫描源，全局源只读复用，并兼容 `BuiltinScanSourceProvider` 扩展
- URL 链接扫描：输入页面链接自动提取页面画像
- 扫描摘要：LLM 驱动的页面画像结构化摘要
- 管理界面：支持创建、编辑、启停、删除项目级扫描源和项目级轨迹规则包，全局配置在项目页只读展示；规则包表单提供模板、结构检查和预览
- 前端加载：页面路由使用 lazy load 拆包，避免开源默认构建出现大 chunk 警告

### 管理后台
- 用户管理：创建、检索、统计
- 模型配置：多模型、多提供商、编辑、删除
- 提示词库：版本管理、审核、内容详情
- TOM 管理：候选审核、批量操作、引用查询
- 资产管理：草稿/正式用例、Skill/Tool 模板

## 架构

```text
backend/          Spring Boot 3.x 后端
frontend-react/   React 18 + TypeScript + Vite 前端
packaging/        打包与部署脚本
```

### 技术栈

- 后端：Spring Boot 3.3.5 + MySQL + Flyway + JDBC
- 前端：React 18 + TypeScript + Tailwind CSS + Vite
- LLM：OpenAI 兼容 API（统一网关、异步任务、错误分类、重试与调用审计）
- 文档解析：PDFBox 3.0.3 + POI 5.2.5

### 数据库

Flyway 迁移文件位于 `backend/src/main/resources/db/migration/`，按版本号递增管理。

## 业务接入

平台不依赖 Java 硬编码业务包类。新业务接入方式：

1. **创建项目**：在项目大厅新建项目
2. **导入资料**：上传使用手册、导入页面链接
3. **采集轨迹**：通过采集器录制真实操作轨迹
4. **自动沉淀**：系统根据项目资产自动生成 business pack draft
5. **人工审核**：确认、合并、拆分业务包
6. **持续演化**：后续使用过程中持续更新

## 开源数据边界

- 仓库不附带项目数据库、业务规则包、业务文档、采集结果、截图、日志或诊断导出。
- `.env`、模型密钥、数据库密码和 JWT 密钥只在部署环境配置；版本库仅保留无密钥的示例配置。
- 默认运行态不加载特定业务资产。新业务能力由项目资料、轨迹、TOM、LLM Wiki 和 Business Pack 在使用过程中逐步沉淀。
